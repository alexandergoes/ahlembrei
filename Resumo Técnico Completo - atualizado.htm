AhLembrei — Documento Técnico Completo

1. Visão Geral

AhLembrei é uma plataforma web de segurança e emergência médica. Permite que usuários cadastrem contatos de emergência, dados médicos e documentos, acessíveis publicamente via QR Code, link direto ou busca por apelido — mesmo sem o celular da pessoa.

  • Domínio: https://ahlembrei.com.br
  • Stack: React 18 + Vite 7 + Tailwind CSS 3 + Express 5 + Supabase (PostgreSQL + Auth + Storage)
  • Hospedagem: Hostinger (Node.js Web App, conectado via GitHub)
  • Versão atual: 1.0.0

2. Arquitetura

3. Módulos / Páginas

3.1 LandingPage (/)

Funcionalidades:

  • Busca por código de emergência (QR Code/URL): extrai UUID da URL, ou busca por apelido via RPC search_by_handle, ou navega direto
  • Busca por apelido: campo @apelido, chama RPC search_by_handle, exibe modal de confirmação
  • Fluxo "Consciente": carrega 2 perguntas de segurança aleatórias (get_random_questions), usuário responde, verifica via RPC verify_security_answer (3 tentativas máx), se passar → navega para /emergency/{id}?mode=conscious
  • Fluxo "Inconsciente": notifica WhatsApp do contato primário → navega para /emergency/{id}?mode=unconscious
  • WhatsApp notification: envia mensagem com localização geográfica do socorrista
  • Footer: "v1.0.0", copyright, links para Termos e Privacidade

3.2 EmergencyPage (/emergency/:userId)

Página pública (sem autenticação). Carrega dados via fetchEmergencyProfile + fetchEmergencyContacts + fetchMedicalRecords.

Seções:

  • Perfil: foto (se houver), nome, idade (calculada do birth_date)
  • Endereço: rua, número, bairro, cidade, estado
  • Convênio: empresa, tipo, número da carteira, telefone
  • Contatos de Emergência: nome, relação, email — telefones ocultos 🔒 até clicar "Notificar Família Agora" (envia WhatsApp + revela telefones)
  • Informações Médicas: tipo sanguíneo, alergias, condições, medicações, observações (só aparece se settings_show_medical = true)
  • Helper form: nome + telefone do socorrista (opcional)
  • Notificar Família: envia WhatsApp com localização Google Maps aos contatos primários

5. RLS Policies (Row Level Security)

profiles

  • "Users can view own profile" → auth.uid() = id
  • "Users can insert own profile" → auth.uid() = id
  • "Users can update own profile" → auth.uid() = id
  • "Anyone can view emergency profile" → USING (true) (chave para funcionamento público)

emergency_contacts

  • "Users can view own contacts" → auth.uid() = user_id
  • "Users can insert own contacts" → auth.uid() = user_id
  • "Users can update own contacts" → auth.uid() = user_id
  • "Users can delete own contacts" → auth.uid() = user_id
  • "Anyone can view emergency contacts" → USING (true)

medical_records

  • "Anyone can view medical records" → USING (true)

user_documents

  • "Anyone can view documents" → USING (true)

emergency_logs

  • "Anyone can insert logs" → USING (true)
  • "Users can view own logs" → auth.uid() = user_id
  • "Admin can view all logs" → check admin role
  • "Admin read all logs" → USING (is_admin())

security_questions

  • Policies: user-scoped CRUD

Storage buckets

  • documents: autenticado upload, owner delete
  • avatars: público read, autenticado upload (pasta por user), owner delete

13. Observações Técnicas

  • O banco PostgreSQL tem RLS ativado em todas as tabelas públicas. As policies "Anyone can view" são essenciais para o funcionamento da página de emergência (anon key precisa ler profiles, emergency_contacts, medical_records).
  • O auth.users é gerenciado exclusivamente pelo Supabase Auth. A trigger handle_new_user() sincroniza com profiles.
  • As migrations 001-005 foram aplicadas via CLI Supabase. Migrations 008-014 precisam ser executadas manualmente no SQL Editor.
  • O server Express roda como SPA fallback: qualquer rota que não seja /api/* serve dist/index.html.
  • Contas órfãs (auth.users sem profiles) podem ocorrer se a trigger falhar. Migration 014 limpa automaticamente.
  • Mercado Pago está em modo sandbox. Trocar token para production quando for para o ar.

para o registro no inpi tem uma pergunta de qual tipo de algoritmo hash é usado no sistema, você pode me dizer e ajudar a responder? as opções são: SHA224, SHA256, SHA512 ou outros se outros qual o nome ?

O algoritmo usado é bcrypt (Blowfish). O sistema usa a extensão pgcrypto do PostgreSQL com gen_salt('bf') + crypt() — 'bf' = Blowfish, o mesmo algoritmo do bcrypt.

Nenhuma das opções SHA (224/256/512) corresponde. A resposta para o INPI é:

"outros — bcrypt (Blowfish)"

Detalhe do código (005_security_questions.sql):

-- Hash (armazenamento):

INSERT INTO ... VALUES (..., crypt(p_answer, gen_salt('bf')))

 

-- Verificação:

RETURN stored_hash = crypt(p_answer, stored_hash);
